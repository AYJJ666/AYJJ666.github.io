一. 进程、线程与并发

  • 进程

    • 指在系统中正在运行的一个应用程序
    • 程序一旦运行就是进程
    • 进程——资源分配的最小单位

  • 线程

    • 是操作系统能够进行运算调度的最小单位。它被包含在进程之中，是进程中的实际运作单位
    • 一条线程指的是进程中一个单一顺序的控制流

  • 进程与线程的关系

    • 线程是程序执行的最小单位，而进程是操作系统分配资源的最小单位
    • 一个进程中可以并发多个线程，每条线程并行执行不同的任务（代码段）
    • 进程之间相互独立，但同一进程下的各个线程之间共享程序的内存空间(包括代码段，数据集，堆等)及一些进程级的资源(如打开文件和信号等)，某进程内的线程在其他进程不可见
    • 调度和切换：线程上下文切换比进程上下文切换要快得多

  • 并发

    • 两个或更多地任务（独立活动，进程）同时发生（进行）
    • 一个程序同时执行多个独立的任务

  • 关于

    • 主线程
      • 一个进程至少包含一个线程，即主线程，主线程是唯一的
      • 主线程与进程拥有共同的生命周期，即进程诞生主线程诞生，进程结束主线程结束
    • 其他线程（子线程）
      • 主动按需创建
      • 有独立的堆栈空间，有自己的生命周期
      • 合理使用线程可以提高程序运行效率，但线程之间的切换需要消耗额外的时间、内存等资源，所以要根据需求创建，并不是越多越好
    • 线程id
      • 每个线程都有自己的编号，称为线程id
      • 可以使用std::this_thread::get_id获取线程id（此函数的返回类型为thread::id）​

二. 创建线程

1. 使用thread简单实现

  • 包含头文件thread

  • 使用的函数

    • thread name(fun*){}

      • thread为标准库中已有的类，name为线程对象名，fun为已有函数（也称为线程入口函数）
      • 无返回值
      • 需要传入线程中执行的函数的指针（fun*)，线程创建后开始执行fun函数，fun函数执行结束则线程执行结束

    • join()

      • join()函数的作用是阻塞调用本函数的线程。
      • 如：在线程A中调用了线程B的B.join()，则A线程执行到B.join();时会停止并等待B线程执行，B线程执行结束后，A线程继续执行。

    • detach()

      • 将子线程与主线程的关联分离。被detach()的线程将转入后台由运行时库接管，主线程失去对该子线程的控制权。
      • 主线程结束时，由运行时库负责清理与子线程相关的资源。

    • joinable()

      判断线程能否被join()或detach()，能返回true，不能返回false。

2. 共享内存

  • 只读的数据：安全稳定，不需要特殊处理

  • 有读有写：比如2个线程写，4个线程读，如果没有特殊处理，程序一定会异常

    • 分析：以容器为例，使用时对容器的读写可能只需要一个语句，但系统实际执行时会分成多个步骤（如指针的移动，数据的读写…)，在多任务环境下，由于不同线程任务的切换，正在执行的读写操作就可能被打断并干扰，造成读写的失败或异常，进而导致整个程序的异常或崩溃。

    • 处理办法：制定规则，不能同时读写——同一时刻只能一个线程读或写

      即锁定所有的读写代码，同时只能有一个执行

四. 互斥量mutex

1. 基本概念

  • 是类对象，包含lock()和unlock()成员函数
  • 可以理解为一把锁，可以锁定一段代码（一般是操作共享空间的代码），被锁定的代码只有在lock()成功的情况下才能继续执行，被同一个互斥量锁定的多个线程，同一时间只有一个能够锁定成功（成功的标志是lock()函数返回），如果锁定失败，线程会卡在lock()处并不断尝试lock()
  • 互斥量的锁要谨慎使用，尽量只锁定需要保护的代码（锁定的代码多了影响效率，少了起不到保护作用）

2. 用法

  • 一般使用步骤

    • lock()—>操作共享数据（执行保护代码）—>unlock()
    • lock()和unlock()要成对使用，有lock()必然要有unlock()，每调用一次lock()，就需要对应地调用一次unlock()
    • 不允许调用1次lock()却调用2次unlock()，或调用2次lock()却只调用1次unlock，这种非对称的调用会导致程序不稳定甚至崩溃（也不允许嵌套使用同一个互斥量的lock()和unlock()函数）

4. 死锁

  • 死锁是指两个或两个以上的进程在执行过程中，由于竞争资源或者由于彼此通信而造成的一种阻塞的现象，若无外力作用，它们都将无法推进下去。此时称系统处于死锁状态或系统产生了死锁，这些永远在互相等待的进程称为死锁进程。

  • 死锁的产生至少需要两个线程和两个互斥量。

  • 例如，有两个互斥量mutex1和mutex2，两个线程入口函数A()和B()，A()和B()中均包含需要mutex1和mutex2同时锁定的代码，但是两个互斥量锁定的顺序不同，如下

5. unique_lock()

（1）unique_lock()取代lock_guard

  • unique_lock是类模板，工作中一般使用lock_guard（推荐使用）

  • unique_lock和lock_guard功能类似，都是帮助管理mutex，但unique_lock更加灵活，相应的，效率低一点，占用内存多一点

（2）unique_lock的第二个参数

  • adopt_lock

    和lock_guard的第二个参数adopt_lock意义相同

  • try_to_lock

    • 使用此参数时，unique_lock会尝试调用互斥量的lock()函数来锁定互斥量，但无论成功与否，都会立即返回。

      不能先lock()，再使用unique_lock的try_to_lock

    • 可使用unique_lock对象的owns_lock()判断是否锁定成功，不同情况执行不同代码

  • defer_lock

    初始化一个没有加锁的mutex（不能自己先lock()），方便实用unique_lock的成员函数

（3）unique_lock的成员函数

  • lock()

    手动加锁，但是只加锁一次时不需要自己解锁

  • unlock()

    （暂时）解锁，需和前面的lock()配对使用

  • try_lock()

    尝试加锁互斥量，成功返回true，失败返回false，不会阻塞线程

  • release()

    返回它所管理的mutex对象的指针，并释放所有权，即解除unique_lock对象和mutex对象之间的关系。

    如果本身处于加锁状态，需自行接管并适时解锁。

（4）粒度
  • 互斥量锁定代码的长度称为粒度
  • 锁定的代码越少，粒度越细，效率越高，因此需要选择合适的粒度
（5）unique_lock所有权的传递

  • unique_lock<std::mutex> uniqueLock(mutex);

    • 此词语中unique_lock()对象uniqueLock拥有对mutex的所有权，可以通过uniqueLock管理mutex

    • 所有权只能转移，不能复制

    • 使用unique_lock<std::mutex> uniqueLock2(std::move(uniqueLock1));即可转移所有权

      转移后uniqueLock1为空

  • 使用对象接收函数返回的临时对象也可传递所有权

五. 单例设计模式

在整个开发中只能创建一个类对象

六. 条件变量condition_variable

  • condition_variable

    是一个类，需要和互斥量配合使用

  • wait()函数

    • 函数原型wait(mutex &mut, bool resule)
    • mut为需要锁定的互斥量，result为一个true/false的值，可以是表达式的结果等
    • wait()的作用。在mut锁定成功的情况下，检查result的值。如果result为true，则wait()直接返回，流程继续；如果result为false，则暂时解锁mut并且阻当前塞线程到本行，直到被notify_one/notify_all唤醒
    • 如果没有第二个参数，效果和第二个参数为false一样，会阻塞线程直到被notify_one/notify_all唤醒

  • notify_one/notify_all

    唤醒一个/所有正在wait的线程（只能唤醒正在wait的线程，如果没有正在wait的线程，则调用是无效的）

    • wait()有第二个参数时

      被唤醒后不断尝试锁定互斥量，锁定成功后检查第二个参数的值，为true流程继续，为false则解锁互斥量并继续阻塞线程

    • wait()没有第二个参数

      没有第二个参数时，第一次执行等同于第二个参数为false，但被唤醒时，无条件等同于第二个参数为true，即首次执行流程阻塞，被notify_one/notify_all唤醒流程可以继续向下执行

七. 使用async创建线程

1. async

是个类模板，用来启动一个异步任务，启动一个异步任务之后，返回一个 future对象

  • 启动异步任务

    自动创建一个线程并开始执行对应的线程入口函数（线程入口函数的传递类似thread）

    受系统调度和默认设置影响，不一定会创建新线程

  • future

    是个类模板，future对象中包含线程入口函数的返回结果（线程的运行结果），可通过future对象的成员函数git()来获取

    future提供了一种访问异步操作结果的机制，这个结果可能不能立刻获取，线程执行完即可获取（结果是将来可以拿到的值）

  • get()

    future对象的成员函数，用于获取线程运行的结果

    • get()只能调用一次，需注意保存结果值
    • 如果调用get()时对应的线程还没有执行完，则get()会起到类似join()的作用，阻塞当前线程直至future对应线程结束获取到结果，然后将结果返回

  • wait()

    作用类似join()，相比于get()它只等待线程结束，不返回结果

  • 相比于thread，async对象不需要join()或detach()，并且future对象析构时会调用wait（即主线程中创建的线程执行完后主线程才会结束）

    注：由于编译器和运行环境的差异，可能不会创建新的线程，关于此内容请继续阅读

2. async的额外参数

我们可以通过向std::async传递额外参数（该参数为std::launch类型，一种枚举类型）来达到一些特殊的控制效果

  • launch::deferred

    • 表示async对象的线程入口函数被延迟调用，只到调用对应future对象的wait()或get()函数时才执行

    • 如果不调用对应future对象的wait()或get()函数，则不会创建线程/执行线程入口函数

    • 调用对应future对象的wait()或get()函数时，不会再创建新的线程，而是在原有线程（主线程）中调用对应的线程入口函数

  • launch::async

    立刻创建线程并执行线程入口函数

  • 不写额外参数时相当于传入launch::async | std::launch::defered

    此时系统会自行决定是异步（创建新线程）还是同步（不创建新线程）方式运行；

3. packaged_task

  • 打包任务，把任务包装起来

  • 是个类模板，模板参数为各种可调用对象，通过std::packaged_task把各种可调用对象包装起来，方便作为线程入口函数使用

  • 打包lambda表达式

  • packaged_task包装的可调用对象还能直接调用，所以packaged_task对象也可视作可调用对象

  • 用容器储存/管理packaged_task对象

4. promise

类模板，可以在某个线程中给它赋值（使用set_value()成员函数），然后在其他线程中取出这个值使用。

5. wait_for() & future_status

  • wait_for()

    future的成员函数，等待一段时间，返回一个future_status对象，通过future_status对象的值反映future对象对应的线程的执行情况

  • future_status

    枚举类型，有三个值

    • ready：表示线程执行完毕（返回）
    • timeout：表示等待一定时间后线程没有执行完（没有返回），即执行超时
    • deferred：表示线程被设置为延迟调用

    以上程序中，调整wait_for的时间可以得到ready和timeout的结果，调整async的额外参数（设置为deferred）可得到deferred的结果（尝试得知，部分环境下async默认设置为延迟调用）

6. shared_future

  • 类模板，和future类似

  • get()函数

    • future的get()函数，由于设计时使用了移动语义，只能调用一次，向多个线程共享同一结果时使用受限
    • shared_future设计用于多个线程共享结果，它的get()函数调用时会复制结果，可多次调用

  • shared_future需要通过future对象构造(前两种写法，都使用移动语义)

    std::shared_future<int> results(move(result));

    std::shared_future<int> results(result.share());

    也可直接接收get_future返回的future对象构造(myThr为已有的packed_task对象)

    std::shared_future<int> result(myThr.get_futere);

    std::shared_future<int> result = myThr.get_future();

  • valid()

    判断future对象中是否存在有效值

    bool ifCanGet = future.valid();

直接获取shared_future

八. 原子操作

  • 问题引入

    C++多线程的核心是互斥量，通过互斥量保护共享数据。即使在多个线程中只对同一个变量进行简单读写操作，也要按照lock()-->操作共享数据-->unlock()的步骤，这样虽然安全，但是执行效率有所降低。因此，引入原子操作。

  • 原子操作

    • 在多线程中不会被打断的程序执行片段，是一种不需要互斥量（无锁）的多线程并发方式
    • 原子操作只适用于简单变量的部分不可分割的（要么完成，要么没做，不存在中间态）操作，相比于互斥量，它不需要频繁加锁解锁，因此效率更高

  • atomic

    • 类模板，用于封装某个类型（内置基本数据类型）的值
    • 不是所有操作都可安全进行，只适用于赋值、++、–、+=等操作（如num = num + 1;会出现问题）
    • 功能单一，多用于多线程中计数
